WHAT THIS RECORDING PROVES
This is the human-verifiable proof of the single-source-of-truth consolidation. For every row
of the rewritten LEGACY_LAYER.md rule-by-rule coverage table, the walkthrough points at the
exact C# rule or service under Algo/Risk/ that now owns it — demonstrating the mandate: define
each rule ONCE (on the canonical RiskLimitSet), then enforce it in BOTH patterns (the per-order
pre-trade gate and the portfolio-wide circuit breaker), OR preserve it as a documented,
intentionally-distinct pair that still shares the one threshold. It also shows the two rules
that used to exist ONLY in SQL (order notional value, daily traded volume) are now first-class
C# rules, and that SQL Server holds no risk thresholds, no accept/reject decisions, and no P&L
math after the refactor.

         Circuit breaker | Pre-trade gate | Status), the merged rows annotated "Merged", the
         preserved-distinct rows annotated as such, and the two former SQL-only rules

[~00:35]  Open Algo/Risk/RiskLimitSet.cs — the single source of truth.       → frame _01
         RiskLimitSet is the canonical threshold value object. Each limit is a property, declared
         exactly once and consumed by BOTH enforcement patterns:

             public decimal? MaxOrderPrice { get; init; }
             public decimal? MaxOrderQty { get; init; }
             public decimal? MaxOrderValue { get; init; }
             public decimal? MaxPositionSize { get; init; }
             public decimal? MaxDailyVolume { get; init; }
             public int?     MaxOrderFreqCount { get; init; }
             public int?     MaxOrderFreqWindowSeconds { get; init; }
             public decimal? MaxCommissionTotal { get; init; }
             public decimal  CommissionRate { get; init; }   // no C# literal default; the DB
                                                             // (dbo.RiskLimits.commission_rate,
                                                             // DEFAULT 0.0005) is the single
                                                             // source, hydrated by the gate (MA-15)

         Two behaviours mirror the RiskLimits table exactly. First, "not enforced" is NULL or a
         non-positive ceiling: IsCeilingEnforced(ceiling) returns true only when ceiling > 0, and
         the EffectiveMax* projections return null when a limit is not enforced (so a burst
         cannot be rejected by a zero limit). Second, the most-specific limit row wins:
         SelectMostSpecific(candidates, portfolioId, securityId) applies the precedence
         portfolio+security > portfolio > security, tie-broken by effective_date DESC — the same
         selection the SQL proc used. The pre-trade gate and the RiskManager circuit breaker both
         read the SAME RiskLimitSet, which is what makes "define once, enforce in two patterns"
         literally true.

[~01:05]  Open Algo/Risk/PreTradeRiskService.cs — the seven-check gate.      → frame _01
         PreTradeRiskService is the C# port of dbo.usp_ValidatePreTradeRisk: the per-order gate
         that evaluates one order before it is accepted and returns accept, or the FIRST failing
         reject reason. It resolves the applicable RiskLimitSet (most-specific row) and short-
         circuits to ACCEPT when the set is unlimited (all ceilings NULL/non-positive) — the
         unlimited-when-all-NULL path carried over from the proc. The result contract is
         PreTradeRiskResult { bool IsValid; string RejectReason } with a static Reject(reason)
         helper; the reject-reason strings are preserved verbatim from the SQL, e.g.
         "Order price 999.0000 meets/exceeds limit 500.0000". The seven checks map one-to-one onto
         coverage rows 1–7 — the first four are shown on this frame.

[~01:40]  Coverage rows 1–4 — merged, new, and the one tightening.           → frame _01
         • Row 1 — Order price ceiling. Gate check 1 rejects "Order price … meets/exceeds limit …";
           circuit breaker is Algo/Risk/RiskOrderPriceRule, reading canonical RiskLimitSet.MaxOrderPrice.
           MERGED (duplicative-by-accident): both reject when price >= limit.
         • Row 2 — Order qty ceiling. Gate check 2 rejects "Order qty …"; circuit breaker is
           Algo/Risk/RiskOrderVolumeRule, reading RiskLimitSet.MaxOrderQty. MERGED: both reject
           when qty >= limit.
         • Row 3 — Order notional value (qty × price). Gate check 3 rejects "Order value …";
           circuit breaker is the NEW Algo/Risk/RiskOrderValueRule, reading RiskLimitSet.MaxOrderValue.
           This was SQL-ONLY (max_order_value) and is now a first-class C# rule: both reject when
           qty*price >= limit.
         • Row 4 — Order frequency. Gate check 4 and Algo/Risk/RiskOrderFreqRule now use the
           canonical ROLLING-COUNT algorithm — COUNT of the prior orders still inside the trailing
           window ("now minus MaxOrderFreqWindowSeconds"), rejecting once that count plus the
           current order reaches MaxOrderFreqCount. The rule keeps its Count / Interval config and,
           to stay "never less strict than a correct rolling count" under out-of-order events,
           tracks a high watermark and treats a strictly-earlier (late) event as a breach. This is
           the first of the two intentional behavioural changes of the refactor, and the only
           at-least-as-strict TIGHTENING (AAP §0.6.1; the second change is the non-positive-ceiling
           = "not enforced" convention, AAP §0.3.1 / MA-16): it is AT-LEAST-AS-STRICT as, and never
           looser than, the old fixed-window bucketing.

[~02:40]  Coverage rows 5–7 + the circuit-breaker-only rules.                → frame _02
         • Row 5 — Resulting position size. One shared threshold, RiskLimitSet.MaxPositionSize,
           two evaluation contexts — PRESERVED DIFFERENT-BY-DESIGN (AAP §0.6.2). The circuit-breaker
           rule Algo/Risk/RiskPositionSizeRule checks the CURRENT position (from a
           PositionChangeMessage): "are we already over?". The gate's check 5 projects the
           HYPOTHETICAL POST-fill position ("would this order put us over?") and rejects with
           "Resulting position … meets/exceeds limit …", because it runs pre-trade. Not force-merged.
         • Row 6 — Cumulative commission. One shared threshold, RiskLimitSet.MaxCommissionTotal
           (+ CommissionRate), two contexts — PRESERVED DIFFERENT-BY-DESIGN (AAP §0.6.2). The
           circuit-breaker rules track the ACTUAL commission post-fill, but from different message
           sources: Algo/Risk/RiskCommissionRule reads a PositionChangeMessage (the
           PositionChangeTypes.Commission value), while RiskOrderCommissionRule and
           RiskTransactionCommissionRule read ExecutionMessage.Commission; the gate's check 6
           ESTIMATES commission pre-fill from historical traded notional × rate and rejects with
           "Estimated cumulative commission … meets/exceeds limit …". Same threshold, different
           basis (and different message source) — they do not agree by construction, and that
           is intentional, not a regression.
         • Row 7 — Daily traded volume. Circuit breaker is the NEW Algo/Risk/RiskDailyVolumeRule
           (an in-stream running total partitioned by portfolio+security, rolled over on the UTC
           day); the gate's check 7 reads the persisted accepted/filled/part-filled volume for the
           current UTC day and rejects with "Daily volume … meets/exceeds limit …". This was
           SQL-ONLY (max_daily_volume) and is now a first-class C# rule, sharing RiskLimitSet.MaxDailyVolume.
         • Row 8 — Circuit-breaker-only. Algo/Risk/RiskPositionTimeRule, RiskPnLRule and
           RiskSlippageRule stay C#-only: they need live state (position age, P&L, slippage) that
           the per-order pre-trade gate does not have, so they have no gate column. Their role is
           unchanged by the consolidation.

[~03:50]  PositionRecalculationService + the canonical wiring.               → frame _03
         Open Algo/Risk/PositionRecalculationService.cs — the C# port of
         dbo.usp_RecalculatePositionOnTrade. Its pure core, Recalculate(existingQty,
         existingAvgPrice, existingRealizedPnl, side, tradeQty, tradePrice), returns
         PositionRecalcResult { Quantity, AveragePrice, RealizedPnl }: the same-side branch
         weighted-averages the entry price in, and the opposite-side branch realizes P&L on the
         closed portion (handling exact close, partial close, and flip). It is the SINGLE entry
         point, invoked exactly once per recorded trade from
         Algo/Storages/Sql/SqlLegacyOrderGateway.RecordTradeAsync inside the gateway-owned
         transaction — recomputing from the full persisted trade set rather than folding a delta —
         so the old trigger/standalone double-count hazard (AAP §0.6.4) is eliminated. Finally the
         wiring that proves single-source-of-truth: RiskManager.ApplyCanonicalLimits(RiskLimitSet)
         seeds the circuit-breaker rules from the very same RiskLimitSet the gate reads, so the two
         patterns can no longer disagree about a threshold.

[~04:40]  Closing caption.                                                   → frame _03
         Every row of the LEGACY_LAYER.md coverage table maps to a real C# symbol under Algo/Risk/:
         the merged price/qty rules (RiskOrderPriceRule, RiskOrderVolumeRule), the two formerly
         SQL-only rules now first-class in C# (RiskOrderValueRule, RiskDailyVolumeRule), the
         rolling-count RiskOrderFreqRule (the one intentional tightening), the preserved-distinct
         position-size and commission pairs (RiskPositionSizeRule; RiskCommissionRule /
         RiskOrderCommissionRule / RiskTransactionCommissionRule), the circuit-breaker-only
         RiskPositionTimeRule / RiskPnLRule / RiskSlippageRule, the canonical RiskLimitSet, and the
         two services PreTradeRiskService and PositionRecalculationService. Each threshold is
         defined once and enforced in both patterns (or preserved as a documented distinct pair);
         SQL Server holds no risk or P&L business logic. The three observable outcomes are
         preserved; the only behavioural changes are the two intentional, documented ones — the
         rolling-count frequency tightening (AAP §0.6.1) and the non-positive-ceiling = "not
         enforced" convention (AAP §0.3.1 / MA-16).
