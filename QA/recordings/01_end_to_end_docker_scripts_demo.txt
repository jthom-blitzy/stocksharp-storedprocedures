The SQL→C# risk-consolidation refactor is demonstrated end-to-end against a LIVE SQL Server. It
preserves the three observable outcomes; behavior is preserved except for two intentional,
documented changes (the order-frequency rule adopts the stricter rolling count, AAP §0.6.1, and a
non-positive ceiling now means "not enforced" under the NULL/0 convention, AAP §0.3.1 / MA-16):
(1) the SQL Server 2022 container comes up healthy;
(2) the four scripts apply in run order 001 → 002 → 003 → 004, after which SQL Server is pure
data storage — zero stored procedures and a single best-effort audit trigger, with no risk
thresholds, no accept/reject decisions, and no P&L math left in SQL; (3) the console demo
reproduces the THREE observable outcomes — an order ACCEPTED within limits, an order REJECTED
by the seeded max_order_price ceiling, and a recorded trade that RECOMPUTES the position. After
the refactor the pre-trade decision is made by the C# PreTradeRiskService (the seven canonical
checks) and the position recompute is done exactly once by the C# PositionRecalculationService —
not by dbo.usp_ValidatePreTradeRisk / dbo.usp_RecalculatePositionOnTrade / trg_Trades_PositionRecalc,
which are gone.

ENVIRONMENT  (commands verbatim from Database/README.md)
--------------------------------------------------------------------------------
Stand up the SQL Server 2022 container on the LOOPBACK interface only (host port 127.0.0.1:14330 →
container 1433), with a per-run secret (review finding MA-13, CWE-668):

    # Per-run secret - not committed, not shown in evidence.
    export MSSQL_SA_PASSWORD="$(openssl rand -base64 24)Aa1!"

    docker run -d --name stocksharp-legacy-sql \
      -e "ACCEPT_EULA=Y" -e "MSSQL_SA_PASSWORD=$MSSQL_SA_PASSWORD" \
      -p 127.0.0.1:14330:1433 \
      mcr.microsoft.com/mssql/server:2022-latest

The 127.0.0.1: prefix publishes the port on loopback only; verify with
`docker port stocksharp-legacy-sql` (must print 127.0.0.1:14330, not 0.0.0.0:14330).

Wait for initialization to finish (health check):

    docker logs stocksharp-legacy-sql | grep "Recovery is complete"

Apply the DDL/CRUD scripts in run order 001 → 002 → 003 → 004 (secret passed via SQLCMDPASSWORD so
it never lands on the command line / process list):

    docker cp Database stocksharp-legacy-sql:/tmp/Database
    for f in 001_Schema.sql 002_StoredProcedures.sql 003_Triggers.sql 004_SeedData.sql; do
      docker exec -e "SQLCMDPASSWORD=$MSSQL_SA_PASSWORD" stocksharp-legacy-sql \
        /opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -C -i "/tmp/Database/$f"
    done

Connection string the C# side resolves (fallback; override via the
STOCKSHARP_LEGACY_SQL_CONNECTION environment variable):

    Server=localhost,14330;Database=StockSharpLegacy;User Id=sa;Password=<sa-password>;TrustServerCertificate=True;

Run the console demo:

    dotnet run --project Samples/08_Misc/03_LegacySqlDemo

Credential/exposure note (MA-13): the SA password is a per-run secret generated above, never a
committed literal, and is not pasted into logs, screenshots, or this transcript. The container is
bound to 127.0.0.1 so the SA endpoint is not reachable from other hosts. For the application/demo,
prefer a least-privilege login scoped to the StockSharpLegacy tables (see Database/README.md)
rather than SA, and for any non-disposable target supply the connection string through
STOCKSHARP_LEGACY_SQL_CONNECTION (backed by a secret manager) instead.

Seeded DEMO limits (Database/004_SeedData.sql) — the portfolio-wide RiskLimits row used throughout:

    max_order_price           = 500.00
    max_order_qty             = 10000
    max_order_value           = 1000000.00
    max_position_size         = 100000
    max_daily_volume          = 250000
    max_order_freq_count      = 5
    max_order_freq_window_sec = 60
    max_commission_total      = 5000.00
    commission_rate           = 0.0005

Reference data after seeding: the DEMO portfolio (currency USD) plus the AAPL and MSFT securities
on NASDAQ.
